## 1. 핵심 시나리오 — "김지원(26세)의 전세 계약 이야기"

### 왜 페르소나 시나리오인가?

심사위원이 우리 서비스를 "머릿속에 그릴 수 있어야" 한다.
기능을 나열하면 잊힌다. **한 사람의 이야기로 관통하면 기억에 남는다.**
발표 전체를 관통하는 하나의 시나리오를 설정하고, 그 안에서 기능이 자연스럽게 등장하게 한다.

---

### 페르소나: 김지원

| 항목 | 내용 |
| --- | --- |
| 나이 | 26세, 직장 2년차 |
| 상황 | 서울 관악구 신림동 빌라 전세 계약 예정 |
| 보증금 | 2억 원 (부모님 지원 + 본인 저축) |
| 고민 | "등기부등본은 떼봤는데, 근저당이 뭔지 모르겠고, 이 집이 안전한 건지 판단이 안 돼" |
| 현재 행동 | 네이버에 "전세 사기 확인 방법" 검색 → 체크리스트 블로그 → 읽어도 결국 판단 불가 |

### 시나리오 전체 흐름 (발표 시 슬라이드 5, 9번에서 활용)

### STEP 1. 서비스 진입 — "이 집, 괜찮을까?"

> 김지원은 하나원큐 앱에서 전세대출을 알아보다가,
**"전세 안전점검"** 배너를 발견한다.
> 
- 진입 경로: 하나원큐 앱 메인 배너 또는 전세대출 신청 과정 중 자동 제안
- **심사위원 관점:** "유입이 자연스러운가?" → 전세대출 고객은 이미 하나은행 앱에 있음. 별도 마케팅 비용 없이 기존 퍼널에서 유입됨.

---

### STEP 2. 매물 정보 입력 — 주소 + 보증금, 딱 2개

> 김지원이 입력하는 것:
> 
> - 주소: **서울 관악구 신림로 ○○길 ○○, ○○빌라 301호** (카카오 주소 자동완성)
> - 보증금: **2억 원**
- 지도에서 위치 확인 → "이 주소가 맞나요?" 한 번 더 컨펌
- **심사위원 관점:** "입력이 너무 적은 거 아닌가?" → 이 2개만으로도 1차 분석이 가능한 게 핵심. 허들을 최소화해야 이탈을 줄인다.

---

### STEP 3. 1차 기본 분석 — 3초 안에 즉시 결과

> 입력 즉시, 화면에 나타나는 것:
> 

```
┌────────────────────────────────────────┐
│  📊 1차 기본 분석 결과                    │
│                                        │
│  전세가율: 82%  → 🟡 지역 평균(74%) 대비 높음│
│  보증 가입 가능성: 🟢 HUG 기준 충족 가능     │
│                                        │
│  ⚠️ 더 정확한 분석을 원하시면               │
│  등기부등본을 업로드해주세요                  │
│  [등기부 업로드하기]                       │
└────────────────────────────────────────┘
```

**분석 로직 (내부):**

1. 국토부 실거래가 API → 해당 동·면적 기준 시세 추정 (약 2.45억)
2. 전세가율 = 보증금(2억) ÷ 시세(2.45억) = **약 82%** → 70~85% 구간 → 🟡
3. HUG 전세보증금반환보증 가입 기준 대입 → 가입 가능 → 🟢
- **심사위원 관점:** "이 데이터 어디서 가져오나?" → 국토부 실거래가 API(공공데이터), 보증기관 공개 기준. 모두 공개 데이터로 구현 가능.
- **심사위원 관점:** "3초 만에 가능한가?" → 1차 분석은 API 1회 호출 + 수치 계산. 복잡한 모델이 아니라 룰 기반 판단.

### STEP 4. 등기부등본 업로드 — 선택이지만, 핵심 차별점

> 김지원은 인터넷등기소에서 700원 주고 발급받은 PDF를 업로드한다.
> 

**AI가 자동으로 읽어내는 것:**

| 추출 항목 | 김지원 매물의 실제 값 | 해석 |
| --- | --- | --- |
| 근저당 설정 | 1건, 1.8억 (○○은행) | 소유자 대출이 시세 대비 높은 편 |
| 가압류 | **1건 존재** (2026.02 설정) | 🔴 최근 가압류 → 소유자 재정 문제 신호 |
| 신탁 등기 | 없음 | 🟢 |
| 소유권 이전 | 2024.03 (2년 전) | 🟢 안정적 |

**파싱 기술:**

- PDF → OCR 텍스트 추출 (등기부는 정형화된 문서라 정확도 높음)
- NLP 구조화: 갑구/을구/표제부 구분 → 근저당·가압류·신탁 자동 태깅
- **파싱 실패 시 폴백:** "자동 인식에 실패했습니다" → 근저당 금액, 가압류 유무, 소유권 이전일 3개만 수기 입력
- **심사위원 관점:** "OCR 정확도는?" → 등기부등본은 인터넷등기소에서 발급되는 디지털 PDF라 OCR 정확도가 매우 높음. 스캔 문서가 아님.
- **심사위원 관점:** "틀리면 어떡하나?" → 파싱 실패 시 수기 입력 폴백 + 모든 결과에 면책 문구 상시 표시.

### STEP 5. 2차 정밀 분석 — 위험 노출 구간을 "금액"으로

> 10초 내에 정밀 분석이 완료된다.
> 

**핵심 산출물 — 위험 노출 구간 계산:**

```
[계산 과정 — 발표에서는 숫자 변화만 보여주고, 계산식은 보조]

시세(감정가 추정): 2.45억
× 해당 지역 평균 낙찰가율: 65~80%
= 예상 낙찰가: 1.59억 ~ 1.96억

선순위 채권 (등기부에서 자동 추출):
  - 근저당 1.8억
  = 선순위 총액: 1.8억

임차인 배분 가능 금액 = 낙찰가 - 선순위
  = (1.59억 - 1.8억) ~ (1.96억 - 1.8억)
  = -0.21억 ~ 0.16억

위험 노출 구간 = 보증금 - 배분 가능 금액
  = 2억 - 0.16억 ~ 2억 - (-0.21억)
  = 1.84억 ~ 2.21억
```

> **결론: 보증금 2억 중 약 1.84억~2.21억이 위험에 노출**
→ 거의 전액 손실 가능성
>

## 2. 심사위원이 품을 수 있는 의문점 + 선제 대응

아래는 발표를 듣는 심사위원이 **"어?"** 하고 생각할 수 있는 포인트와, 그때 우리가 준비해야 할 답이다. PPT에 직접 넣지 않더라도, 발표 흐름 속에서 자연스럽게 막아야 한다.

### 의문 1: "기존 서비스랑 뭐가 다른 거야?"

**발표 중 자연스럽게 막는 방법 (슬라이드 3):**

| 서비스 | 하는 것 | 안 하는 것 |
| --- | --- | --- |
| HUG 전세피해예방시스템 | 사기 의심 매물 정보 제공 | 금액으로 안 알려줌, 금융 연결 없음 |
| KB 전세사기위험도 | A~E 등급 매김 | 얼마나 위험한지 금액 없음, 행동 연결 없음 |
| 서울시 안심전세앱(내집스캔) | 전세가율 기반 점수 | 등기부 분석 없음, 금융 연결 없음 |
| **Hana Safe Lease Passport** | **위험 노출 구간을 "금액"으로 추정** + **즉시 금융 행동 연결** | - |

**핵심 한 문장:** "기존 서비스는 '위험한가?'를 알려주고, 우리는 '얼마나 위험하고, 지금 뭘 해야 하는가?'를 연결합니다."

---

### 의문 2: "그거 낙찰가율 계산기 아니야?"

**선제 방어 (슬라이드 6에서 숫자 보여줄 때):**

단순 계산기와의 차이:

1. 등기부 PDF를 AI가 읽어서 선순위 채권을 **자동 추출**한다 (사용자가 직접 입력 안 함)
2. 보증 가입 가능성까지 **자동 판별**한다 (룰 엔진)
3. 결과에 따라 **보증 가입 / 대출 심사 / 상담 예약**까지 원스톱 연결
4. 계산기는 "숫자를 보여주고 끝". Passport는 **"지금 보증에 가입하세요"까지 연결.**

---

### 의문 3: "MVP로 5일 안에 뭘 보여줄 수 있어?"

**선제 방어 (슬라이드 9):**

| 시연 가능한 것 | 기술 근거 |
| --- | --- |
| 주소 입력 → 전세가율 즉시 계산 | 카카오 주소 API + 국토부 실거래가 API |
| 등기부 PDF 업로드 → 핵심 항목 자동 추출 | OCR + 정규식 기반 파싱 (등기부는 정형 문서) |
| 위험 노출 구간 산출 | 수치 계산 (룰 엔진) |
| 🟢/🟡/🔴 판정 + 리포트 화면 | 프론트엔드 목업 |
| 보증/대출/상담 연결 버튼 | 딥링크 (URL 기반) |

**못 보여주는 것 (솔직하게):**

- 마이데이터 실제 연동 (Phase 2)
- ML 기반 이상 거래 탐지 (Phase 2)
- 카카오톡 공유, PDF 다운로드 (Phase 2)

---

### 의문 4: "AI 분석이 틀리면 책임은?"

**선제 방어 (슬라이드 8 또는 13):**

1. 모든 결과에 **면책 문구 상시 표시**: "본 추정치는 법적 확정 금액이 아니며, 계약 의사결정 참고용 정보입니다."
2. 서비스 성격 자체가 **"의사결정 참고 도구"**이지 "법적 판단 도구"가 아님
3. 파싱 실패 시 수기 입력 폴백 → 기술적 오류에 대한 사용자 보완 장치 존재
4. 발급 이력 해시 기록 → 판정 시점의 무결성 검증 가능

### 의문 5: "수익은 어디서 나와?"

**선제 방어 (슬라이드 11):**

```
[수익 모델 퍼널]

Passport 발급 (무료)
    ↓ 10% 전환
전세대출 실행
    ↓
하나은행 대출 수익

Year 1 추정:
- SOM 2만 건 × 10% 전환 = 2,000건 대출
- 평균 대출금 1.5억 × 금리 마진 0.5% = 건당 75만 원
- 2,000건 × 75만 원 = 약 15억 원

+ Phase 2: 프리미엄 분석(유료), B2B 중개사 구독
```

- **심사위원 관점:** "전환율 10% 근거?" → Passport 발급 시점 = 전세 계약 직전 = 전세대출 수요 시점. 보험 비교 플랫폼의 견적→가입 전환율 8~15% 참고.

---

### 의문 6: "공공 서비스가 같은 걸 만들면?"

**선제 방어 (슬라이드 13):**

오히려 기회다:

- 공공(HUG, 내집스캔)이 **진단 범위를 확대**할수록 → 사용자 인식 ↑ → 우리 서비스 유입 ↑
- 공공은 **금융 행동 연결을 하지 않음** (할 수 없음). 은행만 할 수 있는 영역.
- "공공이 진단을 확대할수록, 우리 전환 퍼널 가치가 올라갑니다."

## 3. 발표 스토리라인 — 10분 흐름 정리

| 시간 | 슬라이드 | 말하는 것 | 심사위원이 느끼는 것 |
| --- | --- | --- | --- |

## 4. 시나리오 변주 — 🟢 / 🟡 케이스도 준비

### 🟢 케이스: 안전한 매물

| 항목 | 값 |
| --- | --- |
| 매물 | 서울 마포구 아파트, 보증금 3억 |
| 전세가율 | 62% → 🟢 |
| 등기 | 근저당 없음, 가압류 없음 → 🟢 |
| 보증 | HUG 충족 → 🟢 |
| 판정 | 🟢 계약 가능 |
| 위험 노출 | 약 0~3,000만 원 (보증 가입 시 거의 0) |
| 액션 | **전세보증금반환보증 가입 → 전세대출 사전 심사** |

> **이 케이스의 의미:** 🟢여도 보증 가입을 안내하고, 전세대출로 자연스럽게 연결. **여기서 하나은행 수익이 발생.**
>

> **이 케이스의 의미:** "안 된다"가 아니라 "이렇게 보완하면 된다"까지 알려줌. 사용자 경험이 긍정적.

## 5. MVP 기능 범위 — "이것만 있으면 서비스가 돌아간다"

| 기능 | MVP에서 하는 것 | MVP에서 안 하는 것 |
| --- | --- | --- |
| 유입 | 앱 배너 + 전세대출 연동 | 외부 플랫폼 제휴 |
| 입력 | 주소 + 보증금 | 마이데이터 연동 (UI만 구현) |
| 등기부 | PDF 업로드 + AI 파싱 | 앱 내 등기 열람 대행 |
| 분석 | 핵심 3변수 (등기·전세가율·보증) | ML 모델 (Isolation Forest 등) |
| 리포트 | 3-Layer + 저장 | 카카오톡 공유, PDF 다운로드 |
| 액션 | 보증·대출·상담 연결 | 대안 매물 탐색 |
| 데이터 | 등기부 캐시 (암호화 저장, 30일 유효) | 블록체인 이력 검증 |

**한 문장:** MVP 필수 6개 기능만으로 "입력 → 분석 → 리포트 → 금융 행동"이 완결된다.

## 6. 참가신청서 작성 가이드 — 이 시나리오를 기반으로

### 1. 주제 선정 배경 (1,000자 이내)

**이 순서로 쓸 것:**

1. 전세사기 피해 현황 수치 1개 (20~30대 피해 70%)
2. 기존 대응의 구조적 한계 (정보는 있지만 금융 판단으로 전환 안 됨)
3. "계약 전"이라는 타이밍의 중요성
4. 하나은행이 이걸 해야 하는 이유 (대출·보증·마이데이터·AI 인프라 보유)

### 2. 아이디어 제안 (1,000자 이내)

**이 순서로 쓸 것:**

1. 서비스 한 줄 정의
2. 김지원 시나리오 요약 (3문장)
3. 차별점: "위험 노출 구간을 금액으로" + "금융 행동 즉시 연결"
4. MVP 범위와 구현 계획 (핵심 3변수 + 공공 API + 룰 엔진)
5. 예상 결과물: 작동하는 프로토타입 (주소 입력 → 분석 → 리포트 → 액션 연결)

### 3. 기대효과 (1,000자 이내)

**이 순서로 쓸 것:**

1. 사용자: 전세 계약 전 위험을 금액으로 인식 → 피해 사전 예방
2. 하나은행: Passport = 전세대출 전환 퍼널 → 수익 창출 (Year 1 약 15억)
3. 사회: 금융포용 (청년 주거 안전망) + ESG(S) 가치
4. 확장성: Phase 2(마이데이터·AI Q&A), Phase 3(기후 리스크·블록체인)
