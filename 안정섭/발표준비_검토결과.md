# 하나 전세안심패스 — 본선 발표 준비 검토 결과

> 분석 모델: Claude + Codex(GPT) + Gemini 3중 교차 검증  
> 발표 조건: 7분, 평가 기준 — 실현가능성(30%) · 사업성(30%) · 참신성(30%) · 자료완결성(10%)  
> 작성일: 2026-04-13

---

## 1. 확정 사항 — 그대로 사용해도 되는 부분

### 1-1. 핵심 차별점 프레이밍
- "HUG=정보, KB=등급, 서울시=점수, Hana=**행동**" 구도 — 변경 불필요
- 클로징 문장으로도 강력: "Hana는 행동을 연결합니다"
- 3개 모델 모두 "설득력 있다"고 동의

### 1-2. 위험 노출 구간 계산 공식
```
예상 낙찰가       = 감정가(시세) × 지역 평균 낙찰가율
임차인 배분 가능  = 예상 낙찰가 - 선순위 채권 총액
위험 노출 구간    = 보증금 - 임차인 배분 가능 금액
```
- 법원경매 실무 원리와 일치 — 공식 논리 확정
- 범위(구간)로 산출하는 방식도 적절

### 1-3. 종합 판정 로직
| 판정 | 조건 |
|------|------|
| 🔴 신중 검토 권고 | 핵심 3변수 중 하나라도 🔴 |
| 🟡 보완 후 가능 | 🔴 없이 🟡 2개 이상 |
| 🟢 계약 가능 | 🔴 없고 🟡 1개 이하 |

- 보수적 설계로 사용자 보호에 타당 — 면책 문구와 함께 유지
- Python 구현 완료(registry_ai_logic.py)로 실현가능성 입증 가능

### 1-4. MVP 기술 스택
- OCR(pdfplumber) + 정규식 NLP + 룰 기반 판정 엔진 — 현실적
- 등기부등본은 인터넷등기소 발급 정형 PDF → OCR 정확도 높음
- "ML 없이 구현 가능"이라는 점이 오히려 실현가능성 신뢰도를 높임

### 1-5. 유입 채널 전략
- 전세대출 신청 과정 중 자동 제안 → 별도 마케팅 비용 없이 기존 퍼널 활용
- 논리적으로 강함 — 변경 불필요

## 2. 추가 조사 필요 — 수치 출처 확보 필수

> 심사위원이 출처를 물었을 때 답변 불가하면 사업성·자료완결성 감점 직결

### 조사 목록

| 수치 | 현재 상태 | 조사 방법 | 우선도 |
|------|-----------|-----------|--------|
| **낙찰가율 65~80%** | 근거 없음 | 대법원 법원경매정보(사이트 직접 조회), KB리서치 부동산 리포트 | 🔴 필수 |
| **전환율 10%** | "보험 비교 앱 8~15%" 비유만 | 금융 앱 유입-대출 전환 관련 업계 리포트, 하나금융 IR 자료 | 🔴 필수 |
| **SOM 2만 건** | 계산식 없음 | 국토부 전월세 거래량 통계 × 수도권 비중 × 하나원큐 도달률 (아래 역산식 참고) | 🔴 필수 |
| **평균 대출금 1.5억** | 출처 없음 | 하나금융 공시, 금감원 전세대출 통계, KB부동산 평균 전세가 | 🟠 중요 |
| **금리마진 0.5%** | 수익 계산에 사용됨 | 하나은행 전세대출 금리 체계 확인, 이자 이익률 공시 자료 | 🟠 중요 |

### SOM 역산식 예시
```
전국 연간 신규 전세 계약 건수 : 약 100만 건 (국토부 전월세 거래량)
× 서울·수도권 비중             : 약 50%  → 50만 건
× 빌라·오피스텔 비중           : 약 40%  → 20만 건
× 하나원큐 사용자 도달 비율    : 약 10%  → 2만 건 (Year 1 SOM)
```
- 위 수치는 추정값 — 실제 통계로 검증 후 확정

### HUG 보증 가입 기준 확인
- 기획안에 "HUG 기준 충족 여부"로만 언급됨
- 실제 조건(전세가율 100% 이하, 보증금 7억 이하 등)이 기술 명세의 판정 로직과 일치하는지 확인 필요
- **Gemini 지적**: 하나손해보험 보증 인수 기준과 서비스 위험 판정이 맞는지 내부 확인 권장

## 3. 변경 필요 — 발표에서 리스크가 되는 부분

### 3-1. 🔴 Neo4j / GraphRAG 언급 위치 통일
**문제**: AI 설계 문서에서는 Neo4j + GraphRAG가 핵심 구조처럼 기술됨. 기능 명세서에서는 Phase 2로 분류. **문서 간 불일치**.

**변경안**:
- 발표 자료에서 Neo4j / GraphRAG 언급 완전 제거
- "MVP 기술 = OCR + 룰엔진 + 수치계산"으로 명확히 통일
- "AI Q&A 기능은 Phase 2에서 온톨로지 기반으로 고도화 예정" 한 줄로만 처리

---

### 3-2. 🔴 F-04 마이데이터 발표에서 제거
**문제**: "동의 UI만 구현, 실제 반영은 Phase 2"는 심사위원 질문 유발.  
"동의만 하고 반영이 안 되면 왜 누르나?" → 설명이 복잡해짐.

**변경안**:
- 발표 자료에서 마이데이터 기능 언급 삭제
- Phase 2 로드맵 슬라이드에 "마이데이터 연동 → 개인 재무 상황 반영 정확도 향상" 한 줄로 흡수

---

### 3-3. 🟠 수익 모델 표현 수정
**문제**: "Year 1 약 15억"은 순이익이 아니라 이자 이익 매출. 개발·운영 비용이 미반영됨.

**변경안**:
```
현재: "Year 1 수익 약 15억"
변경: "Year 1 이자 이익 매출 추정 약 15억 (보수 추정 기준)"
```
- 손익분기점(BEP) 한 줄 추가 권장: "초기 개발 비용 대비 BEP는 Year 2~3 목표"
- 또는 규모를 낮춰 현실적으로 제시: "파일럿 목표 1,000건 → 전세대출 전환 100건"

---

### 3-4. 🟠 등기부 파싱 실패 폴백 UX 개선 명시
**문제**: "파싱 실패 → 수기 입력 3개 항목"은 청년 사용자 이탈 유발 가능.

**변경안**:
- 발표에서 폴백 언급 시 "파싱 실패율은 X% 이하 (디지털 PDF 기준)" 수치 제시
- 또는 "파싱 실패 시 1차 분석(전세가율 + 보증 가능성)만 제공하고 등기 분석은 생략" 대안 언급

---

### 3-5. 🟡 미등기 채권 한계 대응 준비 (Q&A용)
**Gemini 독자 지적**: 차용증·구두 약정 등 미등기 채권은 등기부로 파악 불가.

**발표/Q&A 대응**:
> "등기부 기반 분석은 공시된 선순위 채권만 반영합니다. 미등기 채권은 확인 불가하며, 이 경우 전문 상담 연결로 보완합니다."

---

### 3-6. 🟡 법적·규제 검토 언급 추가 (Codex 지적)
- 금융소비자보호법 적용 여부
- 등기부·신용정보 활용에 대한 개인정보 보호 정책
- "분석 결과는 법적 확정 금액이 아닌 참고용 정보" 면책 문구 → 약점이 아닌 **신뢰 장치**로 프레이밍
